# lesser-host: Evidence Plan (Rubric v0.1.1)

This document defines the **evidence surface** for the lesser-host governance rubric.

Goal: every rubric item maps to **deterministic, reproducible evidence artifacts** written under `gov-infra/evidence/`.

## Source of truth
- Rubric: `gov-infra/planning/lesser-host-10of10-rubric.md`
- Controls matrix (requirements → controls → verifiers → evidence): `gov-infra/planning/lesser-host-controls-matrix.md`
- Threat model (stable threat IDs): `gov-infra/planning/lesser-host-threat-model.md`

## How evidence is generated (canonical)
Evidence is produced by **one deterministic entrypoint**:

```bash
bash gov-infra/verifiers/gov-verify-rubric.sh
```

This verifier:
- runs checks from the repo root (stable working directory)
- writes per-ID logs to `gov-infra/evidence/<RUBRIC-ID>-output.log`
- writes additional deterministic artifacts (e.g., `gov-infra/evidence/coverage.out`)
- emits a single machine report: `gov-infra/evidence/gov-rubric-report.json`

## Evidence file conventions
- `gov-infra/evidence/gov-rubric-report.json`: the summarized, machine-readable rubric result.
- `gov-infra/evidence/<ID>-output.log`: the primary human-readable log for rubric ID `<ID>`.
- `gov-infra/evidence/DOC-5-parity.log`: dedicated log for the threat ↔ controls parity rule.
- `gov-infra/evidence/coverage.out`: Go coverage profile output generated by the verifier.

Verifier outputs are **deterministic paths** so CI can reliably collect them as build artifacts.

## Review/retention guidance (CI)
- CI should run the verifier on protected branches and pull requests.
- CI should upload `gov-infra/evidence/**` as job artifacts.
- Treat any status other than `PASS` as a merge blocker (especially `FAIL` and `BLOCKED`).

## Supply-chain allowlisting (SEC-3)
If the supply-chain scanner emits a high-confidence false positive, allowlist it by adding the **exact finding ID** to:

- `gov-infra/planning/lesser-host-supply-chain-allowlist.txt`

Rules:
- one ID per line
- include a justification as a `#` comment directly above the ID
- prefer fixing/removing the risky dependency over allowlisting

## Evidence integrity expectations
- Evidence is only meaningful if produced by the canonical verifier (not hand-edited).
- If a check cannot be made deterministic (missing pins/tooling), it must report `BLOCKED` rather than silently passing.
